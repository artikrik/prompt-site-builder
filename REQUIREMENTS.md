# **Technical Requirements: Автоматизована SaaS-платформа для AI-генерації B2B-сайтів**

## **1. Огляд проекту (Project Overview)**

Створення автономної SaaS-платформи для масового пошуку лідів (мікробізнес без цифрової присутності) та автоматичної генерації повноцінних SEO-оптимізованих веб-сайтів. Продукт створюється за моделлю "сайт під ключ до моменту продажу" з інтеграцією систем онлайн-бронювання та платіжних шлюзів для забезпечення максимальної конверсії холодних продажів.

## **2. Стек технологій (Technology Stack)**

| Компонент | Технології | Призначення |
| --- | --- | --- |
| **Frontend (Dashboard)** | SvelteKit, TypeScript, Tailwind CSS | Адмін-панель для управління лідами та клієнтський портал. |
| **Backend API** | NestJS або Go, Node.js | Оркестрація мікросервісів, бізнес-логіка, робота з API. |
| **База даних** | PostgreSQL + Prisma (або Drizzle), Redis | Зберігання користувачів, лідів, проектів. Черги завдань (BullMQ). |
| **Генератор сайтів** | **Hugo** (Static Site Generator) | Генерація структурованих сайтів (Markdown, `hugo.toml`, layouts). |
| **Інфраструктура** | Ubuntu, Docker, Docker Compose | 100% Self-Hosted рішення на власному сервері. |
| **Web Server / Proxy** | **Caddy** | Зворотний проксі, роутинг, автоматичний SSL (On-Demand TLS). |
| **CI/CD** | GitHub Actions (Self-Hosted Runner) | Автоматичне тестування та розгортання коду на сервері. |

## **3. Архітектура та бізнес-логіка (Core Architecture)**

Система розробляється за принципами Clean Architecture та складається з ізольованих модулів.

### 3.1. Модуль лідогенерації та скрапінгу

* **Джерело 1 (Google Maps):** Пошук бізнесів за критерієм "відсутній веб-сайт" (інтеграція з Apify/Outscraper API).
* **Джерело 2 (Соціальні мережі):** Збагачення профілю ліда контентом (публікації, опис) через Instagram API парсери.
* **Обмеження:** Строге дотримання GDPR (збір лише публічної B2B-інформації).

### 3.2. Генеративне ядро (AI Generative Core)

Повна відмова від локальних LLM. Управління логікою через патерни стратегії (`ILLMStrategy`, `IImageGenerationStrategy`).

* **Генерація тексту та коду:** Використання виключно хмарних API (Anthropic Claude 3.5 Sonnet або OpenAI GPT-4o) для формування структури Hugo, написання оптимізованих текстів та конфігураційних файлів.
* **Генерація медіа:** Інтеграція з Text-to-Image API (наприклад, DALL-E 3) для створення унікальних фонових зображень (hero images) та візуального контенту. Усі медіафайли програмно зберігаються в директорію `static/images/` проекту Hugo.
* **Управління ключами:** Усі API-ключі зберігаються виключно в `.env` файлах через єдиний безпечний Config Module.

### 3.3. Модуль компіляції та публікації (Site Publisher Service)

* Перехоплює згенеровану структуру файлів від LLM.
* Запускає дочірній процес (child process) `hugo build`.
* Переміщує скомпільовану папку `public/` у визначену директорію на сервері (наприклад, `/var/www/client-sites/salon.pp.ua`).
* Взаємодіє з Caddy для моментальної доступності сайту за унікальним піддоменом.

### 3.4. Сервісні інтеграції (Client Widgets)

* **Бронювання:** Програмна ін'єкція віджетів EasyWeek.
* **Платежі:** Інтеграція API WayForPay / Monobank (plata by mono) для створення інвойсів та прийому оплат.

## **4. Інфраструктура та Web-сервер (DevOps & Hosting)**

* **Локальний хостинг:** Всі компоненти (БД, бекенд, фронтенд, статичні сайти) розгортаються на локальному Ubuntu сервері.
* **Caddy Server:** Використовується як основний веб-сервер та зворотний проксі. Ключова вимога — налаштування **On-Demand TLS** для динамічної генерації SSL-сертифікатів від Let's Encrypt для кожного нового клієнтського піддомену (наприклад, `*.pp.ua`).
* **Контейнеризація:** Платформа повинна запускатися єдиною командою `docker compose up -d`.
* **CI/CD Pipeline:** Налаштовані YAML-файли для GitHub Actions, що тригерять збірку, прогін тестів та деплой через Self-Hosted Runner, встановлений на Ubuntu.

## **5. Вимоги до забезпечення якості (QA & Testing)**

Рівень покриття тестами має відповідати стандартам Senior-інжинірингу. Код не приймається без відповідних тестів:

* **Unit Testing (Vitest/Jest):** Покриття всієї бізнес-логіки бекенду, валідаторів структури Hugo та ізольованих компонентів Svelte.
* **Mocking:** **КРИТИЧНА ВИМОГА.** Усі виклики до зовнішніх платних API (Anthropic, OpenAI, Apify) повинні бути обов'язково замокані у тестовому середовищі для уникнення витрат токенів та забезпечення швидкості CI.
* **Integration Testing:** Перевірка взаємодії між модулями (наприклад, БД -> Сервіс -> API endpoint).
* **E2E Testing (Playwright):** Покриття критичних шляхів користувача (авторизація в дашборді, запуск генерації сайту, перегляд результатів).
* **Статичний аналіз:** Суворі налаштування ESLint, Prettier та TypeScript.
